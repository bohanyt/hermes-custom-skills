## What Happened

Final iteration of Live2Video pipeline (Part 3). Key insight: previous approach applied 5C per short clip, but the correct approach is **one 5C narrative for the entire 10+ minute video** built from selected clips across all topics. ## Problem Identified (from Part 2)

- 22 topics → 22 short scripts → repetitive, unsatisfying
- 5C was applied per-topic instead of for the whole video
- Topic_022 script was underdeveloped with too much repetition

## Correct Approach (v1.3)

Single 5C narrative spanning the entire video:
1. **CONTEXT** (00:00-02:00) — Bohan opens stream, first reaction to "Neverness to Everness" cinematic graphics
2. **CHARACTER** (02:00-04:00) — Discovers character Kokoh, Code Zero incident, absurd dialogue with Director
3. **CONFLICT** (04:00-07:00) — Boss battle Chicken Jacket (intense, cool animations, impact frames), lag issues, game mechanic confusion
4. **CHOICE** (07:00-09:00) — Bohan chooses to explore city, finds anomalies, meets Mbak anomaly hunter, teleport mechanics
5. **CONSEQUENCE** (09:00-10:00+) — Gacha/pack opening (rare character drop), skin reveal, stream winding down

## Technical Work

- **Source video**: Remuxed AV1 MP4 → MKV (stream copy, ~2 min, 5.7GB)
- **44 chunks → 22 topics → 1 unified 5C script** (`script.json`)
- **41/41 clips extracted** using ffmpeg with `-hwaccel cuda` + NVENC
- **Seek bottleneck discovered**: Stream copy from 5.7GB file was slow due to keyframe seek I/O (not encoding). NVENC didn't help because bottleneck was I/O seek, not encoding
- **Concat → Trim → Final concat** approach: first full assembly (38.8min), then trim 32 clips, then final assembly (9m10s)

## Key Technical Learnings

- **Seek I/O bottleneck**: For large source files (5.7GB+), ffmpeg seek time dominates over encoding. `-hwaccel cuda` + NVENC doesn't solve seek latency
- **5C for long-form**: Framework should govern the entire video arc, not per-clip. Per-clip 5C creates repetitive, shallow narratives
- **Batch clip extraction**: Sequential extraction of 41+ clips from large files needs careful timeout management; partial-progress resume pattern works best
